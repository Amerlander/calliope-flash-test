Filename convention
The N in makecode-N-{ble,radio}.hex is the Calliope mini hardware version the hex targets: makecode-1-* → mini v1 (nRF51, legacy DAPLink); makecode-2-* → mini v2 (nRF52833, J-Link OB); makecode-3-* → mini v3 (nRF52833, DAPLink, hardware-identical to micro:bit V2).

How codal decides partial-flash support

codal's MicroBitMemoryMap::findHashes() (in both the legacy nRF51 and v2/nRF52 codebases) scans flash post-app for two magic patterns:

PXT magic

0x923B8E70  0x41A815C6
0xC96698C4  0x9751EE75

16-byte sentinel followed by 8-byte DAL hash + 8-byte MakeCode hash. Embedded by the PXT linker when an editor compiles user code.

uPy magic

0x597F30FE …
0xC1B1D79D

16-byte header at the end of a page; region records sit before the header. Embedded by addlayouttable.py during the codal MicroPython build.

For makecode hexes, PFS is only registered if pxt.json has partial_flashing=1 in optionalConfig. Without that, codal still runs but never advertises the PartialFlashing characteristic. The 2026-05-26 pxt-calliope rebuild added this for all three hardware targets.

The MP region uses HASH_TYPE_POINTER: codal reads the field as a pointer to the version string and CRC32s it at boot. That's how MicroPython detects "different variant" (BLE↔radio) — different release strings → different CRCs → widget triggers full DFU on variant change.

blocks.hex — no magic by design

The codal blocks-runtime expects PXT magic + hashes to be appended by the editor when a user program is compiled and bundled. A bare blocks-runtime build has no magic, so PFS REGION_INFO returns all-zeros. scratch-calliope's model: flash the runtime once over USB, then stream Scratch blocks over MbitMore (0b50f3e4-…) at runtime — re-flashing isn't part of the flow.

Confirmed on hardware: blocks.hex on mini v3 advertises PFS, DFU, and MbitMore in app mode, but every REGION_INFO returns 0x00000000-0x00000000.
