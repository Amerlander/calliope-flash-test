E2E test findings — open-link-test hexes

Tested against a Calliope mini v3 (DAPLink unique-id 9906…c6, nRF52833, hardware-identical to micro:bit V2). Python + bleak for raw probes; Chrome + @calliope-edu/mini-connection-widget@01efeb7 for widget e2e (matches calliope-campus pin).

Filename convention
The N in makecode-N-{ble,radio}.hex is the hardware version the hex targets, not a build iteration. Only mini v3 hexes can be exercised on this rig: blocks.hex, makecode-3-ble.hex, makecode-3-radio.hex, python_ble.hex, python_radio.hex. The makecode-1-* (mini v1) and makecode-2-* (mini v2) files are listed in ANALYSIS for completeness but skipped here.

All five mini-v3 hexes are widget-compatible. Three connect in app mode without intervention; two need A+B+Reset to surface BLE. Every reachable hex exposes the unbonded 8EC90003 characteristic and no bonded 8EC90004 — open-link confirmed throughout.

PFS=False for MakeCode is consistent: pxt-calliope's codal config doesn't enable MICROBIT_BLE_PARTIAL_FLASHING, so the service simply isn't registered. The widget falls back to full DFU for any re-flash.

Region readback — python_ble

codal's partial-flashing service reports regions matching the uPy FlashLayout statically parsed from the hex at 0x66FF0:

RegionStatic (from hex)Live (from device)
0 (SD)0x00001000–0x0001C0000x00001000–0x0001C000 ✓
1 (MP)0x0001C000–0x000674A4, hash=PTR0x0001C000–0x000669A4, hash=1bee72ce ✓ (CRC32 of version string)
2 (FS)0x0006D000–0x000730000x0006D000–0x00073000 ✓

Region 1's hash (HASH_TYPE_POINTER) is the codal-computed CRC32 of "v2.1.2b…", used by the widget to detect "different variant" between BLE and radio MicroPython builds → trigger full DFU on variant change.

Widget e2e — campus widget code path

Widget at HEAD 01efeb7 (matches calliope-campus pin) connected to both BLE-on firmwares in app mode and to the radio variants in pairing mode. From results/widget-session.log:

[info] Connected (BLE)
[info] BLE state: app-mode — partial-flash visible;
       services=[0000180a, 0000fe59, e97dd91d]                   ← python_ble.hex
       services=[0b50f3e4, e97dd91d, 0000180a, 0000fe59]         ← blocks.hex

One full BLE-DFU round trip via the widget (1195 KB image) completed in 19.8 s. The widget's BLE reconnect daemon recovered from a post-flash disconnect in ~43 s.

Partial flashing only works on python_ble.hex out of the box. For partial-flash on blocks.hex, scratch-calliope needs to append PXT magic when bundling user blocks. For makecode-3-ble.hex, pxt-calliope would need to enable MICROBIT_BLE_PARTIAL_FLASHING.

How to reproduce

Plug a Calliope mini v3 in (DAPLink mounts as E:), then from open-link-test/:

# Static analysis on all 9 hexes
python scripts/analyze_all.py

# App-mode probe loop on all 9
python scripts/probe.py

# Pairing-mode probe for the BLE-off-in-app variants
python scripts/probe_paired.py flash python_radio.hex
# A+B+Reset on the mini, hold for full grid fill
python scripts/probe_paired.py probe python_radio_pairing

# Widget e2e
cd ../flash-test/widget-demo
pnpm install                    # one-time
pnpm dev                        # in one terminal
node puppeteer-driver.mjs --park   # in another; click connect in Chrome
