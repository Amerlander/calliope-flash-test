## Filename convention

The `N` in `makecode-N-{ble,radio}.hex` is the **Calliope mini
hardware version** the hex targets:

- `makecode-1-*` ‚Üí Calliope mini v1 (nRF51, legacy DAPLink)
- `makecode-2-*` ‚Üí Calliope mini v2 (nRF52833, J-Link OB)
- `makecode-3-*` ‚Üí Calliope mini v3 (nRF52833, DAPLink,
  hardware-identical to micro:bit V2)

All three hardware variants were tested in this pass ‚Äî see
[FINDINGS.md](FINDINGS.md).

## How codal decides partial-flash support

`MicroBitMemoryMap::findHashes()` in codal-microbit (both the legacy
nRF51 and v2/nRF52 versions) scans flash post-app for **two** magic
patterns:

- **PXT magic** ‚Äî 16 bytes:
  `0x923B8E70 0x41A815C6 0xC96698C4 0x9751EE75`, followed by an 8-byte
  DAL hash + 8-byte MakeCode hash. Embedded by the PXT linker when an
  editor compiles a user program.
- **uPy magic** ‚Äî 16-byte header: `0x597F30FE`, version, table_len,
  num_regions, page_size_log2, `0xC1B1D79D`. Records (SD / MP / FS)
  sit in flash *before* the header. Embedded by `addlayouttable.py`
  during the codal MicroPython build.

If neither magic is present, REGION_INFO returns all-zero ranges and
the widget treats the device as "stub / unsupported partial flash".

For makecode hexes, **PFS is only registered if pxt.json has
`partial_flashing=1`** in `optionalConfig`. Without that, codal still
runs but never advertises the PartialFlashing characteristic. The
2026-05-26 pxt-calliope rebuild added this for all three hardware
targets.

## Python ‚Äî uPy FlashLayout tables

Both python hexes have valid uPy FlashLayout headers at 0x66FF0
(version 1, 3 regions, page_log2=12):

| File | SD region | MP region | FS region |
|---|---|---|---|
| `python_ble.hex` | 0x01000‚Äì0x1C000 (108 KB) | 0x1C000‚Äì0x674A4 (298 KB, PTR `v2.1.2b`) | 0x6D000‚Äì0x73000 (24 KB) |
| `python_radio.hex` | 0x01000‚Äì0x1C000 (108 KB) | 0x1C000‚Äì0x67488 (298 KB, PTR `v2.1.2b-radio`) | 0x6D000‚Äì0x73000 (24 KB) |

The MP region uses `HASH_TYPE_POINTER`: codal reads the field as a
pointer to the version string and CRC32s it at boot. That's how
MicroPython detects "different variant" between BLE‚Üîradio: different
release strings ‚Üí different CRCs ‚Üí widget triggers full DFU on
variant change.

## `blocks.hex` ‚Äî no magic by design

The codal blocks-runtime expects PXT magic + hashes to be appended by
the editor when a user program is compiled and bundled. A bare
blocks-runtime build has no magic, so PFS REGION_INFO returns
all-zeros. scratch-calliope's model is to flash the runtime once over
USB, then stream Scratch blocks over MbitMore (`0b50f3e4-‚Ä¶`) at
runtime ‚Äî re-flashing isn't part of the flow.

Confirmed on hardware: blocks.hex on mini v3 advertises PFS, DFU, and
MbitMore in app mode, but every REGION_INFO returns
`0x00000000-0x00000000`.

## Two delivery patterns

**Pattern A ‚Äî partial image (no bootloader bundled)**: the four
legacy makecode hexes (`makecode-1-*`, `makecode-2-*`) for mini v1/v2.
Stock Nordic MBR at 0x0-0x7C0 (nRF51) or 0x0-0xB00 (nRF52833), SD +
app, no bootloader at 0x78000. Relies on the device having a
bootloader already; UICR record in the hex is ignored by mass-storage
drag-flash on both DAPLink and J-Link OB.

**Pattern B ‚Äî full image (bootloader + MBR included)**: all five
mini-v3 hexes. codal-patched MBR at 0x0-0xB00, SoftDevice at 0x1000,
app at 0x1C000, FS at 0x77000, bootloader at 0x78000-0x7F000, MBR
settings @0x7E000, bootloader settings @0x7F000.

A first-flash of any Pattern B mini-v3 file installs the open-link
bootloader on a fresh device.

## Memory layout ‚Äî mini v3 (Pattern B)

```
0x00000000  codal MBR (patched)                       2,816 B
0x00001000  SoftDevice S140                          ~107 KB
0x0001C000  App (code + magic + hashes/layout)    137‚Äì306 KB
0x00077000  FS / scratch                                4 KB
0x00078000  Open-link bootloader (v0.3.5-campus-open-1) 24 KB
0x0007E000  MBR settings + bootloader settings          5 KB
```

## Memory layout ‚Äî mini v1/v2 (Pattern A, nRF51-style)

```
0x00000000  Nordic MBR (stock)
0x00018000  App + magic + hashes  (note: SoftDevice ends 4 KB earlier
            than nRF52833 ‚Äî smaller SD on nRF51 / older codal builds)
0x00035C00  MakeCode user code page
0x0003BBFF  end of usable flash
```
