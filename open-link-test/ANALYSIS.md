# open-link-test hex file analysis

Analysis of the 9 hex files in this folder, run on 2026-05-26.
Generated by the scripts in [`scripts/`](scripts/).

## Filename convention

`makecode-N-{ble,radio}.hex` — the **N** is the Calliope mini hardware
version the hex targets, not a build iteration:

- `makecode-1-*` → Calliope mini v1 (legacy hardware, **not testable on
  the mini v3 connected to this rig**)
- `makecode-2-*` → Calliope mini v2 (legacy hardware, **not testable on
  the mini v3 connected to this rig**)
- `makecode-3-*` → Calliope mini v3 (nRF52833, hardware-identical to
  micro:bit V2)

Only the **mini v3** hexes can be exercised end-to-end on our hardware:
`blocks.hex`, `makecode-3-ble.hex`, `makecode-3-radio.hex`,
`python_ble.hex`, `python_radio.hex`.

## How codal decides partial-flash support

The codal partial-flashing service
([`MicroBitMemoryMap::findHashes()`](../../LLM/FIRMWARE/codal-microbit-v2/source/MicroBitMemoryMap.cpp))
scans flash post-app for **two** magic patterns:

- **PXT magic** — 16 bytes:
  `0x923B8E70 0x41A815C6 0xC96698C4 0x9751EE75`, followed by an 8-byte
  DAL hash + 8-byte MakeCode hash. Embedded by the PXT linker when an
  editor compiles a user program; used by pxt-calliope (MakeCode) and
  by codal blocks-runtime once Scratch-compiled code is bundled.
- **uPy magic** — 16-byte header: `0x597F30FE`, version, table_len,
  num_regions, page_size_log2, `0xC1B1D79D`. Records (SD / MP / FS)
  sit in flash *before* the header. Embedded by `addlayouttable.py`
  during the codal MicroPython build.

If the firmware finds neither magic at runtime, REGION_INFO responses
over the BLE PartialFlashing service return all-zero ranges → the
widget's defensive guard catches that and falls back to full DFU.

## Detail per mini-v3 hex

### `python_ble.hex` and `python_radio.hex`

Both have valid uPy FlashLayout tables. The MP region uses
`HASH_TYPE_POINTER`: codal reads the field as a pointer to a version
string and CRC32s it at boot. That's how MicroPython detects "different
variant" (BLE ↔ radio) — different `MICROBIT_RELEASE` strings produce
different CRCs → widget triggers full DFU on variant change.

| File | Layout @ | SD region | MP region | FS region |
|---|---|---|---|---|
| `python_ble.hex` | 0x66FF0 | 0x01000–0x1C000 (108 KB) | 0x1C000–0x674A4 (298 KB, PTR `2.1.2b`) | 0x6D000–0x73000 (24 KB) |
| `python_radio.hex` | 0x66FF0 | 0x01000–0x1C000 (108 KB) | 0x1C000–0x67488 (298 KB, PTR `2.1.2b-radio`) | 0x6D000–0x73000 (24 KB) |

### `makecode-3-ble.hex` and `makecode-3-radio.hex`

Both have valid PXT magic at 0x46000. The DAL hash in region 1 lets
the widget distinguish "same runtime, different user code" (→ partial
flash) from "different runtime" (→ DFU). The MakeCode hash in region 2
captures the user code variant.

| File | PXT magic @ | DAL hash (codal runtime) | MakeCode hash (user code) |
|---|---|---|---|
| `makecode-3-ble.hex` | 0x46000 | `f2c18696e64d8403` | `5d002400a300b000` |
| `makecode-3-radio.hex` | 0x46000 | `5260a64ea1db7570` | `f300c200fe00a800` |

DAL hashes differ between BLE and radio variants → BLE↔radio swap
triggers full DFU as expected.

### `blocks.hex`

Has neither PXT nor uPy magic. This is by design: the codal
blocks-runtime expects the PXT magic + hashes to be appended by the
editor (scratch-calliope) when a user program is compiled and bundled.
A bare blocks-runtime build has no magic, so PFS REGION_INFO returns
all-zero ranges.

In practice the scratch-calliope flow flashes the runtime once over
USB, then streams Scratch blocks over MbitMore at runtime instead of
re-flashing — so flash-side partial flash isn't part of the model and
the missing magic is harmless.

## Two delivery patterns

**Pattern A — partial image (no bootloader bundled)**: the
`makecode-1-*` and `makecode-2-*` files. Stock Nordic MBR at 0x0-0x7C0,
SoftDevice + app only, no bootloader at 0x78000. Relies on the device
having a bootloader already; UICR record in the hex is ignored by
DAPLink mass-storage flash.

**Pattern B — full image (bootloader + MBR included)**: all five
mini-v3 hexes. codal-patched MBR at 0x0-0xB00, SoftDevice at 0x1000,
app at 0x1C000, FS at 0x77000, bootloader at 0x78000-0x7F000, MBR
settings @0x7E000, bootloader settings @0x7F000.

A first-flash of a Pattern B file installs the open-link bootloader on
a fresh mini.

| `makecode-3-ble.hex` | `0x20000400` | `0x00000A81` | `0x20020000` | `0x00036A41` |
| `makecode-3-radio.hex` | `0x20000400` | `0x00000A81` | `0x20020000` | `0x00037019` |
| `python_ble.hex` | `0x20000400` | `0x00000A81` | `0x20020000` | `0x00028775` |
| `python_radio.hex` | `0x20000400` | `0x00000A81` | `0x20020000` | `0x0002718D` |

All Pattern B files share the codal MBR stack pointer (`0x20000400`)
and reset vector (`0x00000A81`) — same MBR build. App vector tables
at 0x1C000 all show SP at the top of nRF52833 SRAM (`0x20020000`,
i.e. the 128 KB mark, since the codal app stack grows down into the
SRAM block starting at 0x20020000) and valid Cortex-M Thumb reset
vectors (low bit set, addresses in flash).
